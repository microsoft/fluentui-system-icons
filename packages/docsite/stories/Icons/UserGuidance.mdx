<h1 className="sbdocs-title">User Guidance</h1>

`@fluentui/react-icons` supports three rendering strategies: **inline SVG** (default), **icon fonts**, and **SVG sprites** (⚠️ Alpha). Each has different trade-offs for performance, caching, and styling flexibility.

> **Tip:** Regardless of which rendering approach you choose, always use **atomic imports** for best tree-shaking and bundle size. See the [Atomic API](../?path=/docs/icons-atomic-api--docs) page for details.

> **Note:** A **base** rendering mode (vanilla CSS, no Griffel runtime) is available as an alpha opt-in via `/base/*` imports (e.g. `@fluentui/react-icons/base/svg/access-time`). With CSS extraction it reduces JS bundle sizes significantly — up to −70% for font icons, −56% for SVG sprites, −14% for inline SVG. This table will be updated once the base API ships in stable.

### When to use inline SVG (default)

This is the recommended approach for most applications:

- Zero configuration needed — works out of the box
- Best styling flexibility — supports `primaryFill`, `className`, Griffel, and all CSS properties
- Excellent tree-shaking — only icons you import are bundled
- Combine with the **Atomic API** for even smaller bundles (see the Atomic API page)

Use inline SVG unless you have a specific reason to choose another approach.

### When to use icon fonts

Consider icon fonts when your application renders **many icons simultaneously** (e.g., data grids, file explorers, dashboards):

- **Rendering performance**: Font glyphs render faster than individual SVG DOM nodes when hundreds of icons are on screen
- **Reduced DOM complexity**: Each icon is a single `<i>` element instead of an `<svg>` tree
- **Caching**: Font file is downloaded once and cached by the browser

**Accessibility concerns:**

- **Screen readers may misinterpret glyphs.** Icon fonts render characters from the Unicode Private Use Area. Screen readers may announce the underlying character name or code point instead of the icon's meaning. Always pair icon font elements with `aria-label` or `aria-hidden="true"` with a visible text alternative.
- **Pseudo-elements are not reliably exposed to assistive technologies.** Icon fonts often rely on `::before` / `::after` with `content` to render glyphs. These pseudo-elements are inconsistently announced across screen readers and browser combinations.
- **Glyph interpretation varies across environments.** Different operating systems, font rendering engines, or encoding mismatches can cause glyphs to display incorrectly or as blank squares. SVG icons are immune to this class of issues.

**Font loading issues (FOIT & FOUT):**

Icon fonts are subject to the same loading problems as any web font:

- **FOIT (Flash of Invisible Text)** — icon glyphs are invisible until the font file loads. The browser hides text during the font's load period, causing elements to briefly disappear.
- **FOUT (Flash of Unstyled Text)** — fallback system font glyphs are shown first, then "flash" to the correct icon font glyphs once loading completes. This causes layout shifts and a jarring visual swap.

**Mitigation:** Base64-inline the font into a critical CSS block. This eliminates the loading flash by embedding the font data directly in the render-blocking stylesheet, at the cost of increased CSS size.

**Other trade-offs:**

- Font files are loaded upfront and include all glyphs used in your application, potentially increasing initial load time
- Slight reduction in visual fidelity, though generally imperceptible at most sizes
- Requires additional Webpack configuration (see setup below)
- Styling is limited to CSS properties that work with text (color, font-size) — no `primaryFill` prop

#### Setting up icon fonts

Fluent icons can be consumed as a font. The API is the same as the React element icons, and you can subset the font to include only the glyphs your application uses with the `@fluentui/react-icons-font-subsetting-webpack-plugin` package.

If `optimization.usedExports` is enabled (as it is by default), this plugin will subset the font files to only include the glyphs actually used by your build.
Here is how you set up your webpack configuration to use font files in your application:

```js
// webpack.config.js
const {
  default: FluentUIReactIconsFontSubsettingPlugin,
} = require('@fluentui/react-icons-font-subsetting-webpack-plugin');

module.exports = {
  module: {
    rules: [
      // Treat the font files as webpack assets
      {
        test: /\.(ttf|woff2?)$/,
        type: 'asset',
      },
    ],
  },
  resolve: {
    // Include 'fluentIconFont' to use the font implementation of the Fluent icons
    conditionNames: ['fluentIconFont', 'import'],
  },
  plugins: [
    // Include this plugin
    new FluentUIReactIconsFontSubsettingPlugin(),
  ],
};
```

You can also enable font icons without the subsetting plugin by only adding the `resolve.conditionNames` configuration. However, this is **not recommended** as it will include the full font files for all icon variants.

### When to use SVG sprites (⚠️ Alpha)

> **Alpha prerelease:** SVG sprites support is available as an alpha prerelease only. Install via `npm i @fluentui/react-icons@alpha`. Breaking changes may occur before general availability.

Consider SVG sprites when:

- **Runtime icon loading**: Icons are fetched on demand and cached by the browser, rather than bundled into your JS
- **Dynamic icon sets**: Your application loads different icons depending on context (e.g., CMS-driven UI)
- **Color icon gradient issues**: Sprites avoid the SVG gradient ID collision that occurs when multiple instances of the same color icon exist on a page (see the Color Variants page)

**Trade-offs:**

- Individual HTTP requests per icon (mitigated by browser caching)
- Sprites must be served from the same origin — see the [same-origin requirement docs](https://github.com/microsoft/fluentui-system-icons/blob/main/packages/react-icons/docs/preview-features/svg-sprites.md#same-origin-requirement) for details

SVG sprites are available via the `@fluentui/react-icons` package using atomic sprite imports — the API mirrors the standard Atomic API:

```tsx
import { AccessTime24Filled } from '@fluentui/react-icons/svg-sprite/access-time';

function MyComponent() {
  return <AccessTime24Filled />;
}
```

For setup, tooling, and the same-origin requirement, see the full [SVG Sprites documentation](https://github.com/microsoft/fluentui-system-icons/blob/main/packages/react-icons/docs/preview-features/svg-sprites.md).

## Using multiple colors for single-color icons

To apply multiple colors to an icon (e.g., one color for the fill and another for the outline), layer the `Filled` and `Regular` variants using absolute positioning:

```tsx
import { makeStyles, mergeClasses } from '@fluentui/react-components';
import { ShieldLock48Filled, ShieldLock48Regular } from '@fluentui/react-icons';

const useStyles = makeStyles({
  icon: {
    position: 'absolute',
  },
  filled: {
    zIndex: 1,
    color: 'yellow',
  },
  regular: {
    zIndex: 2,
    color: 'red',
  },
});

function MyComponent() {
  const styles = useStyles();
  return (
    <span>
      <ShieldLock48Filled className={mergeClasses(styles.icon, styles.filled)} />
      <ShieldLock48Regular className={mergeClasses(styles.icon, styles.regular)} />
    </span>
  );
}
```

This technique positions both icon variants in the same space using z-index stacking. The `Regular` variant (z-index: 2) sits on top, providing the outline, while the `Filled` variant (z-index: 1) sits underneath, providing the fill. By applying different colors to each layer, you can create custom multi-color effects.

## Color variants (deprecated)

`Color` icon variants are **deprecated** and should not be used in new code. They remain available for backward compatibility, but we strongly recommend migrating to `Filled` or `Regular` variants instead.

### Why color variants are deprecated

#### 1. High Contrast Mode (HCM) non-compliance

Color variants do not work in Windows High Contrast Mode, making them inaccessible for users who depend on high-contrast settings ([#951](https://github.com/microsoft/fluentui-system-icons/issues/951)).

**Workaround:** If you must keep color variants during migration, bundle them with `Filled` variants and switch using a `forced-colors` media query:

```tsx
import { makeStyles } from '@griffel/react';
import { bundleIcon, iconFilledClassName, iconRegularClassName } from '@fluentui/react-icons';
import { CodeBlock48Color, CodeBlock48Filled } from '@fluentui/react-icons';

const CodeBlock48ColorFixed = bundleIcon(CodeBlock48Filled, CodeBlock48Color);

const useStyles = makeStyles({
  icon: {
    [`& .${iconFilledClassName}`]: {
      display: 'none',
    },
    [`& .${iconRegularClassName}`]: {
      display: 'inline',
    },

    '@media (forced-colors: active)': {
      [`& .${iconFilledClassName}`]: {
        display: 'inline',
      },
      [`& .${iconRegularClassName}`]: {
        display: 'none',
      },
    },
  },
});

function MyComponent() {
  const styles = useStyles();
  return <CodeBlock48ColorFixed className={styles.icon} />;
}
```

**Note:** This approach increases bundle size as both variants are included.

#### 2. SVG gradient ID collision

Color icons with gradients use non-scoped `id` attributes. When multiple instances of the same color icon exist on a page, hiding one with `display: none` removes the gradient definition from the global DOM namespace, which breaks all other instances referencing that ID ([#936](https://github.com/microsoft/fluentui-system-icons/issues/936)).

**Workarounds:**

- **SVG sprites** (recommended): Wrap the icon in a `<symbol>` and reference it via `<use>`, avoiding duplicate gradient IDs.
- **Absolute positioning off-screen**: Use `position: 'absolute', top: '-9999px'` instead of `display: 'none'`.
- **`visibility: 'hidden'`**: Hides the icon while keeping the gradient definition accessible (maintains layout space).

#### 3. Dark theme contrast issues

Color variants have insufficient contrast ratios in dark themes, failing WCAG accessibility standards:

- **Non-text elements** (simple icons): Minimum 3:1 contrast ratio required ([WCAG 2.1 SC 1.4.11](https://www.w3.org/WAI/WCAG21/Understanding/non-text-contrast.html))
- **Text-like elements** (icons with letters/numbers or small details): Minimum 4.5:1 for normal text, 3:1 for large text ([WCAG 2.0 SC 1.4.3](https://www.w3.org/WAI/WCAG21/Understanding/contrast-minimum.html))

Most color icons contain small visual details that could be treated as text-equivalent elements requiring 4.5:1 contrast, making them non-compliant in dark themes.

### Migration path

Replace `Color` variants with `Filled` or `Regular` variants. If you need multi-color effects, use the layering technique described in the **Using multiple colors** section above.
